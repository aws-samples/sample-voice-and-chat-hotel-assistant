# Virtual Assistant Chat Agent Evaluation

Evaluates the hotel chat agent across 7 Amazon Bedrock models using the Strands Evals SDK.

## Prerequisites

- Deployed infrastructure (VirtualAssistantStack + HotelPmsStack)
- AWS credentials configured with Bedrock model access
- Python dependencies installed: `cd packages/virtual-assistant && uv sync`

## Running the Evaluation

```bash
cd packages/virtual-assistant
uv run python eval/run_evaluation.py
```

## Metrics Tracking

The evaluation framework automatically tracks:
- **Token usage**: Input/output tokens per turn
- **Cost**: Calculated cost in USD based on Bedrock pricing
- **Response time**: Time taken for each agent response

See [METRICS_GUIDE.md](METRICS_GUIDE.md) for detailed information on:
- Understanding token usage and costs
- Analyzing performance metrics
- Optimizing for cost and speed
- Using the metrics analysis tools

### Quick Metrics Analysis

After running evaluations, analyze the results:

```bash
# Analyze latest results
uv run python eval/analyze_metrics.py

# Analyze specific results
uv run python eval/analyze_metrics.py eval/results/20260224_185245
```

## What Gets Evaluated

### Single-turn tests (tc-001 to tc-003)
Knowledge queries in Spanish and English verified against the hotel knowledge base.
- **OutputEvaluator**: Factual accuracy of the response
- **TrajectoryEvaluator**: Tool calling efficiency (should use `query_hotel_knowledge` without unnecessary calls)

### Multi-turn tests (tc-004 to tc-009)
Full reservation flows driven by an `ActorSimulator` that plays the customer role.
The simulated customer checks availability, picks a room, provides name/email, and expects a confirmation number.
- **GoalSuccessRateEvaluator**: Binary pass/fail — did the agent complete the reservation end-to-end?
- **ConversationOutcomeEvaluator**: Reviews full conversation for quality, language compliance, and accuracy

## Models and Judges

The 7 models under evaluation are the agent's backbone model. The evaluator judges and the ActorSimulator both use the Strands default BedrockModel (Claude Sonnet) unless you change `JUDGE_MODEL` in `run_evaluation.py`.

## Results

Results are saved to `eval/results/<timestamp>/` with:
- Per-model JSON files with detailed scores, reasoning, and conversation logs
- `summary.json` with aggregated scores across all models
- `aggregate_metrics.json` with cost and performance metrics
- `{model}_conversations.json` with detailed turn-by-turn metrics
- `metrics_analysis.json` (generated by analyze_metrics.py)

## Test Cases

Defined in `eval/test-cases.jsonl`. Each line has a `type` field:
- `single_turn`: Knowledge query with expected output and expected trajectory
- `multi_turn`: Reservation flow with `task_description` in metadata for the ActorSimulator

## Adding Test Cases

```json
{"id": "tc-006", "type": "single_turn", "input": "...", "expected_output": "...", "expected_trajectory": ["tool_name"], "scenario": "...", "metadata": {...}}
{"id": "tc-007", "type": "multi_turn", "input": "...", "expected_output": "", "expected_trajectory": [], "scenario": "...", "metadata": {"task_description": "...", ...}}
```
